# 评估报告生成平台 QA 系统介绍与演示手册

适用场景：面向同事进行一轮系统 QA、进度说明、疑问解答和下一步需求收集。

建议形式：互动式演示，边讲边操作，边收集意见。不要只讲“系统能做什么”，要让同事看到“哪些信息从哪里来、如何影响正文、哪里需要人工判断、哪些点还需要继续建设”。

## 1. QA 会议目标

1. 让同事理解当前系统进度：已经具备 Excel 导入、附件/OCR 证据池、权属正文生成、确价测算正文生成、热区引用跳转、报告预览、Word 下载和归档能力。
2. 解释系统边界：系统负责“自动填充、草稿生成、证据引用、格式后处理”，但估价判断、资料真实性、最终结论仍需估价人员确认。
3. 集中收集需求：字段是否够用、默认话术是否符合业务习惯、附件识别是否覆盖真实资料、报告模板格式是否满足出具要求。
4. 明确后续优先级：优先修影响正式生成的缺陷，其次完善高频业务场景，再考虑打包部署和多人协作。

## 2. 推荐会议流程

| 时间 | 环节 | 操作方式 | 目标 |
|---|---|---|---|
| 5 分钟 | 开场问题 | 先问大家“现在做一份报告最耗时/最容易错的是哪一步？” | 建立 QA 关注点 |
| 10 分钟 | 系统全景 | 展示输入、处理、输出三段流程 | 让大家先有整体地图 |
| 35 分钟 | 逐页演示 | 按第一至第四部分逐字段讲，穿插提问 | 解释字段意义和业务影响 |
| 20 分钟 | 核心功能演示 | Excel 导入、OCR 证据池、热区跳转、基准地价资料、报告预览/下载 | 证明系统进度 |
| 15 分钟 | 现场试错 | 故意留空关键字段、点击橙色“请填写”、切换方法组合 | 暴露可用性问题 |
| 15 分钟 | 需求收集 | 按“必须修/应该做/可优化”三类记录 | 形成下一步清单 |

## 3. 一句话介绍系统

这是一个本地运行的土地估价报告生成平台。它把 Excel 测算表、权属/规划/基准地价附件、人工确认字段和估价方法判断，汇总成可预览、可下载、可归档的 Word/PDF 报告。

## 5. 互动式演示脚本

### 5.1 开场

建议口播：

> 今天不是验收会，而是一次业务 QA。请大家重点看三件事：字段够不够、系统生成的话术能不能直接改成正式口径、哪些地方仍然需要人工确认或新增资料来源。

互动问题：

1. 现在做报告时，最容易漏填的是哪个字段？
2. 哪些附件最常见、最希望系统自动识别？
3. 哪一段报告正文最需要自动生成或自动引用依据？

### 5.2 演示主线

1. 打开系统首页，介绍顶部按钮：底色切换、OCR 证据池、基准地价资料、报告生成及预览。
2. 展示左侧四个页签：报告基础、规划条件、土地权属、确价测算。
3. 拖入 Excel 测算表，说明系统会解析字段、记录来源、回填表单。
4. 上传或粘贴一段权证/规划条件/基准地价资料，说明证据池不会直接覆盖字段，而是生成“待确认注入项”。
5. 到第三部分，切换权属情形和用途类型，生成权属正文；进入热区透视，点击引用跳转。
6. 打开基准地价侧边栏，填写基准地价资料；回到正文点击“请填写基准地价批准机关”等热区，说明跨侧边栏定位。
7. 到第四部分，勾选估价方法，调整资料条件，展示方法预警、确价正文和结果正文。
8. 打开报告预览，刷新渲染，展示 PDF 预览和 Word 自动下载/手动下载。
9. 打开 `03_Result`，说明正式归档目录和临时文件清理机制。

### 5.3 现场试错建议

| 试错动作 | 预期现象 | 收集点 |
|---|---|---|
| 清空基准地价批准机关 | 正文出现橙色“请填写”提示，点击可跳到基准地价侧边栏 | 提示颜色是否明显、跳转是否符合习惯 |
| 只勾选资料不足的方法 | 出现方法适用性预警 | 预警是否太严格或太宽松 |
| 修改自动生成正文 | 字段变为人工修改，后续同步不应轻易覆盖 | 是否需要“恢复系统草稿”按钮 |
| 上传多份附件 | 证据池出现多条待确认项 | 是否需要批量规则或附件分类优化 |
| 生成报告 | PDF 预览并自动下载 Word | 字体、字号、标题、表格、提示高亮是否满足正式报告 |

## 6. 页面功能说明

### 顶部工具区

| 功能 | 作用 | QA 关注点 |
|---|---|---|
| 底色切换 | 在浅色/深色背景之间切换，便于不同投影环境演示 | 下拉框、热区、输入框在两种底色下是否清晰 |
| OCR 证据池 | 上传或粘贴附件文本，提取可注入字段 | 识别字段是否准确，是否需要新增附件类型 |
| 基准地价资料 | 集中维护基准地价文件、文号、批准机关、基准日、更新依据 | 是否符合真实资料收集习惯 |
| 报告生成及预览 | 打开 PDF 预览和 Word 下载区域 | 是否需要更明显的下载入口和生成状态 |

### 通用交互

| 功能 | 作用 | QA 关注点 |
|---|---|---|
| 字段来源标识 | 标记字段来自 Excel、OCR、系统生成还是人工修改 | 来源提示是否够明确 |
| 脏数据保护 | 人工改过的正文不被普通同步轻易覆盖 | 是否需要更细的冲突提示 |
| 热区透视 | 将正文拆成可点击引用块，快速定位来源字段 | 是否所有“请填写”都能跳转 |
| 亮蓝定位 | 点击热区后目标字段闪烁提示 | 是否足够明显 |
| 橙色“请填写” | 对报告缺口做可视化提示 | 是否醒目但不刺眼 |
| 自动草稿/重置草稿 | 普通同步保留人工修改，强制重置会覆盖 | 是否需要二次确认或版本历史 |

## 7. 前端表单字段全量说明

说明：下面按当前前端代码盘点，共 147 个字段。其中有些是直接可见字段，有些是派生字段或模板辅助字段，但 QA 时都建议说明其作用，避免同事误以为“多余”或“不知道从哪里来”。

| 字段 | 代码名 | 意义与作用 | QA 关注点 |
|---|---|---|---|

| 自定义出让目的 | `transfer_purpose` | 当目的不是内置选项时手填，参与项目名称和估价目的表述。 | 是否需要更多标准目的模板。 |

## 8. 需求收集模板

建议现场用这张表记录，不要只记录“有问题”，要写清影响和优先级。

| 类型 | 记录内容 | 示例 |
|---|---|---|
| 必须修 | 影响报告生成、字段错误、无法下载、跳转失败、正式报告格式错 | “最终总价未进入正文” |
| 应该做 | 高频业务需要，但可临时人工处理 | “新增工业仓储用途话术” |
| 可优化 | 体验、提示、默认值、颜色、交互顺序 | “证据池批量注入后希望停留原页” |
| 待确认 | 需要业务负责人定口径 | “未采用方法是否必须逐项说明” |

建议问题：

1. 哪些字段在真实项目中不会填，是否可以隐藏或改成高级项？
2. 哪些字段必须从附件自动识别，不能依赖手填？
3. 哪些默认值属于通道项目特例，不能作为通用默认？
4. 第三部分三段正文是否符合正式报告语言习惯？
5. 第四部分方法预警是否会误伤合理方法？
6. Word 版式是否还有字体、字号、表格、标题问题？
7. 最终交付时大家更想要“本机软件包”“内网网页”还是“服务器多人版”？

## 9. 到其他电脑临时运行方案

### 方案 A：最推荐，演示电脑本机运行

适合你把项目拷到另一台电脑，在那台电脑上打开浏览器演示。

前置条件：

1. Windows 电脑。
2. Python 3.11。
3. 已安装 Microsoft Word 或 WPS，便于 PDF 预览；没有也可以下载 Word，但 PDF 预览可能降级。
4. 已复制整个 `D:\评估报告工具` 项目目录，尤其是 `01_Source`、`02_Process`、`src`、`frontend/dist`。

操作步骤：

```powershell
cd D:\评估报告工具
powershell -ExecutionPolicy Bypass -File .\scripts\install_demo_deps.ps1 -PythonExe python
powershell -ExecutionPolicy Bypass -File .\scripts\start_demo_server.ps1 -PythonExe python
```

浏览器打开：

```text
http://127.0.0.1:8000
```

如果演示电脑上 Python 命令不是 `python`，可改成实际路径，例如：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\start_demo_server.ps1 -PythonExe "C:\Users\Lenovo\AppData\Local\Programs\Python\Python311\python.exe"
```

### 方案 B：一台演示电脑运行，同事通过局域网访问

适合会议室里你用一台电脑开服务，其他同事用自己的电脑访问。

操作步骤：

```powershell
cd D:\评估报告工具
powershell -ExecutionPolicy Bypass -File .\scripts\start_demo_server.ps1 -PythonExe python -Lan
```

脚本会显示类似：

```text
LAN demo URL: http://192.168.1.23:8000
```

同事在同一局域网内打开这个地址即可。

注意：

1. Windows 防火墙可能拦截 8000 端口，需要允许 Python/uvicorn 访问专用网络。
2. 当前前端 API 已改为相对路径，局域网访问时不会再打到同事电脑自己的 `127.0.0.1`。
3. 上传附件和生成文件都发生在运行服务的那台电脑上，注意资料保密。

### 方案 C：需要现场改代码或改前端

如果 QA 过程中需要改页面并即时看效果，需要安装 Node.js。

```powershell
cd D:\评估报告工具
python -m pip install -r requirements.txt

cd frontend
npm install
npm run dev
```

另开一个窗口启动后端：

```powershell
cd D:\评估报告工具
python -m uvicorn src.api:app --host 127.0.0.1 --port 8000
```

浏览器打开 Vite 提供的地址，通常是：

```text
http://127.0.0.1:5173
```

当前 `frontend/vite.config.js` 已配置 `/api` 代理到 `127.0.0.1:8000`。

### 无网络电脑的准备方式

如果演示电脑不能联网，先在有网电脑准备依赖包：

```powershell
cd D:\评估报告工具
python -m pip download -r requirements.txt -d wheelhouse
```

把项目目录和 `wheelhouse` 一起拷过去，在演示电脑安装：

```powershell
cd D:\评估报告工具
python -m pip install --no-index --find-links .\wheelhouse -r requirements.txt
```

## 10. QA 后输出物建议

会议后建议整理三份结果：

1. `QA问题清单`：明确缺陷、复现步骤、责任模块、优先级。
2. `字段调整清单`：新增字段、隐藏字段、默认值调整、附件识别来源。
3. `下一阶段路线图`：打包部署、模板定版、多项目支持、更多附件类型、报告审校规则。

## 11. 当前建议的下一步优先级

1. 用真实项目资料跑 3 到 5 个完整样本，重点核对最终 Word。
2. 把字段按“必填/选填/高级/系统派生”重新分组，降低页面认知负担。
