# App.vue 乱码修复记录与编码规范

## 1. 事件背景

`frontend/src/App.vue` 在历史编辑中混入了 **19 处 mojibake（乱码）**。成因是文件在 UTF-8 与 GBK 之间被错误地重编码/复制，导致中文字符被解码为错误的字节序列。乱码既影响可见文案，也破坏了部分功能逻辑（模板插值、正则匹配、toast 前缀）。

## 2. 修复结论（已验证）

| 项目 | 结论 |
| --- | --- |
| 乱码总数 | 19 处，全部修复 |
| 功能性乱码 | 5 处：模板插值 `图${...}`、正则 `/^图\d+ .../`、toast 前缀 |
| 可见文案乱码 | 14 处：`区 / — / 项 / 率 / 值 / 将 / 或 / ❌` 等字符 |
| 残余项（良性、无需处理） | L1 一个 UTF-8 BOM；L4094 / L6920 的 `?` 是合法正则量词，非乱码 |

### 验证手段与结果

1. `node tmp/scan_pollution3.js frontend/src/App.vue` → `TOTAL: 0`
2. 广覆盖独立扫描（U+FFFD、C1 控制符 U+0080–U+009F、PUA U+E000–U+F8FF）→ 可疑行 0
3. `npm run build`（vite build）→ 成功，12 modules transformed，0 错误
4. SFC 编译 0 错误；`HelloWorld.vue` / `main.js` 扫描干净

> 复核日期 2026-06-18：CH-1 独立重跑上述 1–3 项，结论一致，乱码数 = 0，构建通过。

## 3. 功能性乱码的危害（重点）

可见文案乱码只影响观感，**功能性乱码会导致运行时逻辑错误**：

- **模板插值** `图${index}`：乱码后变量名/前缀错位，渲染出错误的图片标题（caption）。
- **正则** `/^图\d+\s.../`：乱码导致匹配失效，照片编号识别、caption 重排逻辑失灵。
- **toast 前缀**：乱码使提示文案错误，影响用户对操作结果的判断。

因此修复后必须做**运行时复测**（照片 caption 重排、toast 显示），不能只看构建是否通过。

## 4. 根因分析

1. 源文件以 UTF-8 存储，但某次编辑/粘贴在 GBK 环境下进行，保存时按错误编码写回。
2. Windows 默认终端/编辑器编码（GBK/CP936）与项目 UTF-8 不一致，放大了风险。
3. 缺少提交前的编码校验，乱码长期潜伏未被发现。

## 6. 编码规范速览

1. **统一 UTF-8（无 BOM 优先）**：所有源码、模板、文档一律 UTF-8。已存在的良性 BOM 可保留，但新文件不要写入 BOM。
2. **终端强制 UTF-8**：PowerShell 操作前执行 `chcp 65001`，并设置 `$OutputEncoding` 与 `[Console]::OutputEncoding` 为 UTF-8。
3. **提交前必扫**：用 `tmp/scan_pollution3.js` 或等效脚本扫描改动文件，乱码数必须为 0。
4. **改动中文/模板后必构建**：`npm run build` 必须通过，SFC 0 错误。
5. **功能性文本改动需运行时复测**：涉及模板插值、正则、提示文案的修改，要在浏览器实际验证。
6. **禁止用 `echo`/重定向写中文文件**：用编辑器工具按 UTF-8 写入，避免 shell 编码污染。
