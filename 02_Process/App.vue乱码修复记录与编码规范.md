# App.vue 编码事故复盘与防复发规范

> 事故日期：2026-06-18
>
> 影响文件：`frontend/src/App.vue`
>
> 事故类型：整文件错误回写、mojibake、不可逆字符丢失、恢复遗漏导致运行时白屏

## 1. 事故摘要

一次 PowerShell `Set-Content` 整文件回写跨越了 UTF-8 与系统默认编码边界。文件中的中文、emoji、全角标点和部分语法字符被错误解码后再次写入，形成大面积 mojibake；部分字符进一步丢失为字面量 `?`，无法仅靠反向编码还原。

事故并非最初判断的“少量 emoji 和标点乱码”：

- 当前文件约有 300 条可疑行；
- 267 条可通过干净检查点按代码骨架和上下文一一恢复；
- 其余约 33 条主要是事故当天新增代码，需要从事故前补丁恢复；
- Vue 构建曾在第 2656 行附近因引号或标签结构损坏失败；
- 文本和语法恢复后，生产构建虽然通过，但首屏仍因 `form.land_area_mode` 定义遗漏而白屏。

最终恢复不仅修复了可见乱码，还补回了恢复过程中遗漏的：

- `FIELD_REGISTRY.land_area_mode`
- `form.land_area_mode`

## 2. 影响

### 2.1 文本层

- 中文被错误显示为 `锛`、`銆`、`鈥`、`馃` 等典型 mojibake 序列；
- emoji、数学符号和全角标点被污染；
- 部分内容丢失为 `?`，原始字节已不可逆。

### 2.2 语法层

- 字符串结束引号丢失；
- HTML/Vue 标签尖括号或属性引号损坏；
- 模板插值、正则、toast 文案等功能性文本被破坏；
- 编译器只在遇到首个语法断点时停止，不能一次列出所有损坏点。

### 2.3 运行时层

恢复候选文件通过 SFC 编译和 `npm run build` 后，8001 页面仍一片空白。浏览器控制台显示：

```text
TypeError: Cannot read properties of undefined (reading 'value')
```

开发服务器源码定位到模板对 `form.land_area_mode.value` 的访问。结构恢复时模板引用被保留，但状态定义和字段注册被漏掉，因此 Vue 在首帧渲染时崩溃。

结论：**构建通过不代表恢复完整，运行时验证是编码事故恢复的必选项。**

## 3. 根因分析

### 3.1 直接根因

使用 `Set-Content` 对大型中文源码进行整文件回写时，读取、PowerShell 字符串、控制台和写入编码没有形成可验证的 UTF-8 闭环。错误解码后的文本被覆盖回正式文件，原文件字节随即丢失。

### 3.2 放大因素

1. 没有先在临时副本 dry-run。
2. 没有在写入前保存带时间戳备份和哈希。
3. `App.vue` 体积大，模板、状态和业务逻辑集中，单次事故影响面很广。
4. `last_turn_App_vue_changes.patch` 同时包含事故前有效修改和事故后人工修补，不能整体正向或反向套用。
5. 第一轮判断过度依赖肉眼和少量乱码 token，低估了完整中文段落、引号和标签结构的损坏。
6. 第一轮验收止于扫描、SFC 编译和生产构建，没有进行浏览器首屏与控制台检查。
7. 终端显示可能自身发生解码偏差，屏幕上“看起来乱码”不能单独证明文件字节损坏。

## 4. 哪些做法不可靠

### 4.1 整体套用混合补丁

事故前有效修改与事故后修补具有不同语义。整体正向应用会重复或污染修复，整体反向应用会撤销真实业务代码。必须先按事故时间边界拆分补丁。

### 4.2 只堆显式 token 映射

token 映射适合可逆、重复的 mojibake，但无法可靠恢复：

- 已丢失为 `?` 的字符；
- 缺失的引号和尖括号；
- 被整行覆盖或删除的状态定义；
- 当天新增、旧检查点中不存在的代码。

### 4.3 只看典型乱码字符

不存在 `锛/銆/馃` 不代表文件安全。语法可能已损坏，完整中文也可能被替换成另一段“看起来合法”的文本。

### 4.4 只跑构建

构建能发现语法错误，但不保证：

- Vue 模板引用的状态都已定义；
- 首屏渲染不会访问 `undefined`；
- API 调用和交互仍然工作；
- 恢复后的业务语义与事故前一致。

### 4.5 相信终端显示而不检查字节

PowerShell 控制台、管道和字体都可能造成显示偏差。判断编码必须基于严格 UTF-8 解码、文件哈希、十六进制/BOM 信息和 Git diff，而不是只凭屏幕观感。

## 5. 本次有效恢复方法

### 5.1 冻结正式文件

停止继续编辑正式文件，先创建只读备份：

```text
tmp/App.vue.before-final-apply-20260618-142244.bak
```

所有自动恢复均先写入临时候选文件。

### 5.2 划定事故边界

55 个补丁按 `Set-Content` 事故边界拆分：

- 前 19 个：事故前有效修改，可作为恢复源；
- 后 36 个：事故后人工修补，只能作为证据，不能整体重放。

### 5.3 建立恢复源优先级

1. 事故前的同版本干净源码或检查点；
2. 事故前补丁中的新增代码；
3. 干净构建产物或历史 bundle 中可确认的文本；
4. 当前损坏文件保留的变量名、标签、属性和代码骨架；
5. 人工语义判断，仅用于没有可靠原文的最后少量内容。

### 5.4 按代码骨架恢复

通过变量名、标签名、属性名、缩进和邻接上下文匹配干净源码，只替换文本与受损符号，不回退事故后新增业务逻辑。

第一轮结构匹配安全覆盖 267 条可疑行；旧检查点没有的 33 条由事故前补丁补齐。

### 5.5 编译器驱动修复

对临时候选文件循环执行 Vue SFC 解析和编译，根据精确行号修复遗漏的引号、标签和表达式。每次只处理首个确定错误，避免扩大修改范围。

### 5.6 多级验收

按以下顺序验收：

1. 严格 UTF-8 解码；
2. U+FFFD、C1、PUA 和典型 mojibake 扫描；
3. Vue SFC 解析与脚本编译；
4. 模板引用与 `form` 定义差集检查；
5. `npm run build`；
6. 浏览器新会话打开 8001；
7. 页面快照非空；
8. 控制台 0 error；
9. 关键业务测试通过。

完成全部验收后，才用候选文件替换正式文件。

## 6. 强制防复发规则

### 6.1 写入规则

1. 含中文的现有源码禁止使用 `Set-Content`、`Out-File`、`>`、`echo` 或 shell heredoc 做整文件回写。
2. 小范围人工修改优先使用结构化编辑器或 `apply_patch`。
3. 确需程序化重写时，必须：
   - 明确使用 UTF-8；
   - 写入临时文件；
   - 严格解码并扫描临时文件；
   - 比较 diff 和文件规模；
   - 通过编译与运行时检查；
   - 最后原子替换正式文件。
4. 修改前必须保存备份或确保 Git 中存在可恢复提交。
5. 不允许在未验证的情况下覆盖唯一原件。

### 6.2 恢复规则

1. 发现乱码后立即停止继续写入。
2. 先判定是显示问题、可逆 mojibake、U+FFFD、字面量 `?`，还是结构删除。
3. `?` 和 U+FFFD 视为可能不可逆，必须寻找外部干净源。
4. 混合补丁必须先按事故边界分类，禁止整体套用。
5. 所有恢复先作用于临时候选文件。
6. 自动修复不得猜测业务文本；无可靠来源时保留清单并人工确认。

### 6.3 验收规则

对中文源码或模板的修改，至少执行：

```powershell
node .cursor/skills/encoding-mojibake-guard/scripts/check-text-integrity.mjs frontend/src/App.vue
node .cursor/skills/encoding-mojibake-guard/scripts/check-vue-form-contract.mjs frontend/src/App.vue
node tmp/check_vue_candidate.js frontend/src/App.vue
Set-Location frontend
npm run build
```

涉及首屏、模板状态或运行时文本时，还必须用真实浏览器验证页面和控制台。

## 7. 完成定义

编码事故只有同时满足以下条件才算关闭：

- [ ] 原始证据、备份和事故边界已保存；
- [ ] 正式文件可被严格 UTF-8 解码；
- [ ] 自动扫描无错误级命中；
- [ ] 可疑字符已逐条解释或消除；
- [ ] SFC/语言编译器无错误；
- [ ] 模板与状态引用完整；
- [ ] 生产构建通过；
- [ ] 关键测试通过；
- [ ] 浏览器首屏正常；
- [ ] 浏览器控制台 0 error；
- [ ] Git diff 只包含预期修改。
