# 评估报告自动化填充工具项目规范文档 (V1.0)

## 1. 项目背景与目标
本项目旨在通过 Python 自动化技术，实现从“黄璞测算表”到“土地估价报告”的数据自动化迁移。核心目标是解决复杂逻辑判定（如土地征收场景）、数据全生命周期溯源以及填充过程的零误差。

## 2. 目录结构与职责划分
项目采用“输入-处理-输出”三段式解耦结构：

```text
d:\评估报告工具\
├── 01_Source/                # [输入层] 原始数据
│   ├── 01_Excel_Sheets/      # 存放原始测算表（如：黄璞测算表_v1.xlsx）
│   └── 02_Word_Templates/    # 存放带占位符的 Word 模板（由原报告修改而来）
├── 02_Process/               # [处理层] 中间状态与配置
│   ├── config.yaml           # 字段映射配置文件（定义 Excel 坐标与变量名的对应关系）
│   └── mapping_logs/         # 记录每次生成的字段映射快照，用于溯源
├── 03_Result/                # [输出层] 最终成果
│   └── YYYYMMDD_项目名/      # 按日期和项目分包存放生成的报告
├── src/                      # [代码层] 核心模块
│   ├── main.py               # 程序入口
│   ├── extractor.py          # Excel 数据提取模块
│   ├── generator.py          # Word 模板渲染模块
│   └── utils.py              # 通用工具（日志、路径处理）
├── logs/                     # [监控层] 运行时日志
├── requirements.txt          # 依赖清单
└── .gitignore                # 排除 venv、临时文件等
```

## 3. 文件命名规范 (Naming Conventions)
所有文件必须遵循以下格式，禁止使用“副本”、“最终版”等模糊词汇：
- **源代码**: 采用小写字母+下划线（如 `data_extractor.py`）。
- **原始数据**: `[来源]_[描述]_[版本/日期].xlsx`（如 `HP_测算表_20240507.xlsx`）。
- **生成报告**: `[项目编号]_[土地位置]_[生成时间].docx`（如 `TD2024_城南街住宅_20240507_1430.docx`）。

## 4. 数据溯源机制 (Traceability)
为了确保每一份报告都能追溯到原始测算表，系统必须满足：
1.  **指纹记录**: 在生成的 Word 报告末尾（或作为隐藏元数据）自动写入：
    - 使用的原始 Excel 文件名及 MD5 校验码。
    - 运行填充程序的版本号。
    - 生成时间戳。
2.  **映射快照**: 每次生成报告时，同步在 `02_Process/mapping_logs/` 下生成一个 JSON 文件，记录该次填充的所有变量及其对应值。

## 5. 开发与运行环境规范
- **Python 版本**: 统一使用 Python 3.9 或以上。
- **虚拟环境**: 必须在 `venv` 下运行，严禁污染全局环境。
- **异常处理**: 
    - 若 Excel 关键字段缺失，程序必须中断并抛出明确错误（如 `KeyFieldError: 面积单元格(C5)为空`）。
    - 严禁静默失败，所有警告必须记录到 `logs/`。

## 6. 复杂逻辑判定规范 (Condition Logic)
针对“有/无土地征收”等分支，严禁在主流程代码中写死硬编码，需遵循：
- **逻辑配置化**: 在 `config.yaml` 中定义逻辑触发条件。
- **模板标签化**: 使用 `docxtpl` 的 Jinja2 语法处理：
    - `{% if land_status == '征收' %}` ... 征收文案 ... `{% endif %}`
- **模块化文案**: 大段落文案差异应存放在独立的 `.txt` 或 `yaml` 配置中，避免污染主代码。

## 7. 质量保证 (QA)
- **模拟运行 (Dry Run)**: 程序支持调试模式，仅在控制台打印映射结果而不生成文件。
- **单元测试**: 针对测算表读取模块，编写测试用例确保不同格式的测算表（如合并单元格）能正确读取。

## 8. 项目演进与溯源原则
- **代码可追溯**: 使用 Git 进行版本管理。
- **结果可审计**: 任何对生成的报告进行的人工微调，都应在后续版本中通过修改模板或逻辑来自动化，保持“单向生成”原则。
